Обзор системы

Secure Data Exchange Server - это высокопроизводительный асинхронный веб-сервер, предназначенный для безопасного обмена данными между различными системами в корпоративной среде. Сервер обеспечивает полный цикл обработки запросов с многоуровневой системой безопасности и расширенным мониторингом.

🎯 Основное назначение

Сервер предназначен для интеграции различных информационных систем через единый безопасный API-интерфейс с поддержкой:

  • Двухфакторной аутентификации (токен + цифровая подпись)
  • Шифрования передаваемых данных
  • Детального логирования всех операций
  • Гибкой настройки прав доступа для разных эндпоинтов

Архитектурные особенности:

  • 🔐 Многоуровневая безопасность - Комбинированная аутентификация по токену и RSA-подписи с временными метками
  • 📊 Трехуровневое логирование - Одновременная запись логов в базу данных, файлы и консоль с маскированием конфиденциальных данных
  • 🩺 Комплексный мониторинг - Детальные health-чекеры для всех компонентов системы
  • ⚡ Асинхронная архитектура - Высокая производительность за счет асинхронной обработки запросов
  • 🌐 Гибкая CORS политика - Настраиваемая политика кросс-доменных запросов
  • 🔍 Verbose режим отладки - Подробное логирование всех входящих/исходящих запросов для отладки

Установка и запуск

Системные требования:

  • Python 3.7+ - интерпретатор Python с поддержкой асинхронного программирования
  • MSSQL Server 2012+ - сервер баз данных Microsoft SQL Server (опционально, можно запускать без БД)
  • OpenSSL - для генерации RSA ключей безопасности
  • 1 ГБ RAM - минимальный объем оперативной памяти
  • 100 МБ - свободное место на диске

Зависимости Python:

# Основные зависимости (устанавливаются автоматически):
aiohttp >= 3.8.0 # Асинхронный веб-сервер
pyodbc >= 4.0.0 # Подключение к MSSQL
cryptography >= 3.0 # RSA шифрование и подписи

Процесс запуска:

# 1. Клонирование или копирование файлов сервера
# 2. Настройка конфигурационного файла config.json
# 3. Генерация RSA ключей безопасности
# 4. Запуск сервера

# Обычный режим (базовое логирование)
python3 server.py config.json

# Подробный режим (полное логирование запросов/ответов)
python3 server.py config.json --verbose

# Проверка статуса конфигурации без запуска
python3 server.py --status

⚠️ Важные замечания:

  • Перед первым запуском обязательно настройте файл config.json
  • Убедитесь, что пути к ключам безопасности указаны корректно
  • Для продакшн-среды установите "debug": false
  • Рекомендуется использовать --verbose только для отладки

Конфигурация сервера

Все настройки сервера определяются через единый JSON-файл конфигурации. Файл должен содержать все обязательные секции и параметры.

Структура конфигурационного файла:

{ "host": "192.168.0.19", # Хост для прослушивания "port": 8000, # Порт сервера "debug": false, # Режим отладки "security": { ... }, # Настройки безопасности "database": { ... }, # Настройки базы данных "logging": { ... }, # Настройки логирования "cors": { ... } # Настройки CORS }

📝 Формат конфигурации

Конфигурационный файл использует стандартный JSON формат с поддержкой комментариев (нестандартное расширение). Все пути могут быть как абсолютными, так и относительными относительно рабочей директории сервера.

Система безопасности

Сервер использует комбинированную систему безопасности, сочетающую статическую аутентификацию по токенам и динамическую верификацию цифровых подписей с временными метками.

🛡️ Компоненты безопасности

  • Bearer Token Authentication - статическая проверка предварительно согласованных токенов
  • RSA Digital Signatures - динамическая проверка подлинности запроса с использованием асимметричного шифрования
  • Timestamp Validation - защита от replay-атак через временные метки
  • Endpoint Security Levels - гибкая настройка уровня безопасности для каждого эндпоинта

Эндпоинты состояния

Эндпоинты для комплексного мониторинга состояния сервера и диагностики проблем. Предназначены для систем мониторинга, DevOps-инженеров и администраторов.

GET /health

Назначение: Основной эндпоинт проверки жизнеспособности системы. Возвращает сводную информацию о состоянии всех компонентов сервера, включая версию, время работы, статус БД, настройки безопасности и логирования.

Уровень безопасности: Публичный (не требует аутентификации)

Использование: Рекомендуется для health-чеков балансировщиков нагрузки и систем мониторинга.

Структура ответа:

GET /health/security

Назначение: Детальная диагностика системы безопасности. Эндпоинт анализирует текущий запрос, проверяет корректность заголовков безопасности, предоставляет информацию о состоянии ключей и диагностические сообщения.

Уровень безопасности: Публичный (не требует аутентификации)

Особенности: Выполняет глубокий анализ заголовков Token и Signature, даже если они присутствуют в публичном запросе.

GET /health/database

Назначение: Мониторинг состояния подключения к базе данных. Проверяет доступность БД, возвращает информацию о сервере, версию СУБД и диагностические данные о подключении.

Уровень безопасности: Публичный (не требует аутентификации)

GET /health/logging

Назначение: Отслеживание состояния системы логирования. Предоставляет информацию о текущих настройках логирования, путях к файлам логов, уровнях детализации и статусе различных обработчиков логов.

Уровень безопасности: Публичный (не требует аутентификации)

GET /health/network

Назначение: Сетевая диагностика и информация о конфигурации. Отображает текущие сетевые настройки, CORS политику, список доступных эндпоинтов и их URL.

Уровень безопасности: Требует токен аутентификации (уровень 'token')

Примечание: Этот эндпоинт требует аутентификации, так как раскрывает внутреннюю структуру API.

GET /health/cloud

Назначение: Мониторинг состояния облачного хранилища. Проверяет доступность облачного сервиса, возвращает информацию о конфигурации и статусе подключения.

Уровень безопасности: Публичный (не требует аутентификации)

Особенности: Проверяет доступность облачного хранилища через аутентификацию и получение токена.

GET /help

Уровень безопасности: Публичный (не требует аутентификации)

API эндпоинты

Основные рабочие эндпоинты для выполнения бизнес-логики приложения. Все эндпоинты этой группы требуют полной аутентификации (токен + цифровая подпись).

POST /document/list

Назначение: Получение списка документов пользователя.

Уровень безопасности: Требует токен и подпись (уровень 'signature')

Бизнес-логика: Используется для получения полного списка документов, связанных с пользователем, включая их статусы подписания.

Запросы к базе данных: Выполняется вызов хранимой процедуры DOC_Select_ID с параметром @USR_ID, которая возвращает список документов пользователя в формате JSON. Процедура извлекает данные из таблицы документов, включая идентификатор, название, дату создания, описание и статус подписания.

Параметры запроса (JSON body):

Параметр Тип Обязательный Описание
user_id string Да Идентификатор пользователя

Структура данных документа:

Поля документа: Каждый документ содержит полную информацию о себе в системе.

  • id: Уникальный идентификатор документа
  • name: Название документа
  • link: Ссылка на документ
  • date: Дата создания документа (может быть null)
  • desc: Описание документа (может быть null)

🎯 Безопасность эндпоинтов

Параметр endpoint_security позволяет настроить индивидуальный уровень безопасности для каждого эндпоинта:

Уровень Описание Требования Примеры использования
public Публичный доступ без аутентификации Нет требований к заголовкам Health-check эндпоинты, документация
token Требуется только валидный Bearer токен Заголовок Token: Bearer <token> Внутренние API, мониторинг

"endpoint_security": { "health": "public", "health/security": "public", "health/database": "public", "health/logging": "public", "health/network": "public", "help": "public" }

ℹ️ Примечания по безопасности эндпоинтов

  • Эндпоинты проверяются по нормализованному пути (нижний регистр, без начального/конечного слэша)
  • Если правило для эндпоинта не найдено, применяется стандартная безопасность (токен + подпись)
  • Для вложенных путей проверяются родительские правила (например, для /help/info будет проверено правило /help)
  • Все эндпоинты API по умолчанию требуют токен и подпись, если не указано иное

🚫 Блокировка пользователей по эндпоинтам

Параметр endpoint_userblocked позволяет указать список эндпоинтов, для которых должна применяться локальная блокировка пользователя:

Параметр Описание Поведение Примеры использования
endpoint_userblocked Список путей эндпоинтов, для которых включена проверка блокировки пользователя Если путь найден в списке, перед обращением к БД выполняется проверка блокировки через ensure_user_request_not_blocked() Логин, обновление пользователя, платежи, документы, список билетов
Путь отсутствует в списке Для данного эндпоинта блокировка пользователя не применяется Даже если пользователь находится в локальной блокировке, endpoint продолжит обработку запроса Служебные, публичные или исключенные из блокировки маршруты
Нормализация путей Сравнение выполняется без учета регистра, начальных и конечных слэшей /USER/LOGIN, user/login и /user/login/ считаются одним и тем же путем Упрощает конфигурирование и уменьшает риск ошибок в настройке

Пример конфигурации из кода:

"endpoint_userblocked": [ "/user/by-phone", "/user/update", "/ticket/list", "/user/login", "/document/signed", "/document/list", "/user/mailing", "/payment/set", "/payment/calculate-distribution", "/document/load" ]

ℹ️ Примечания по блокировке пользователей

  • Эндпоинты сравниваются по нормализованному пути: в нижнем регистре, без начального и конечного слэша
  • Если endpoint присутствует в endpoint_userblocked, сервер применяет проверку блокировки пользователя до обращения к БД
  • Если endpoint отсутствует в endpoint_userblocked, блокировка пользователя для него не действует
  • Параметр управляет только применением блокировки пользователя, но не заменяет настройку общей безопасности через endpoint_security
  • Для специальных эндпоинтов, например /user/update, логика блокировки может учитывать дополнительные условия, реализованные в коде обработчика

🔄 Механизм работы подписей

Система использует двустороннюю аутентификацию с RSA подписями:

Клиентская подпись (входящие запросы):
  1. Клиент создает строку: token.expiry_timestamp
  2. Шифрует строку своим приватным ключом с PKCS1v15 padding
  3. Кодирует результат в Base64
  4. Отправляет в заголовке Signature
  5. Сервер расшифровывает подпись приватным ключом сервера
  6. Проверяет токен и время истечения
Серверная подпись (исходящие ответы):
  1. Сервер создает строку: token.expiry_timestamp
  2. Шифрует строку публичным ключом клиента с PKCS1v15 padding
  3. Кодирует результат в Base64
  4. Добавляет в заголовок Signature ответа
  5. Клиент расшифровывает подпись своим приватным ключом

ℹ️ Важные особенности

  • Все ответы сервера содержат заголовки Token и Signature
  • Подпись гарантированно добавляется даже к ошибкам аутентификации
  • Для публичных эндпоинтов токен извлекается опционально
  • Система поддерживает несколько алгоритмов padding для совместимости

⚠️ Рекомендации по безопасности

  • Никогда не устанавливайте disable_certificates, disable_token_auth и disable_signature в true в продакшн-среде
  • Регулярно обновляйте список allowed_tokens и меняйте токены
  • Используйте разные токены для разных клиентских систем и окружений
  • Храните приватные ключи в защищенном месте с ограниченным доступом
  • Для критичных эндпоинтов всегда используйте уровень 'signature'
  • Настройте соответствующий signature_ttl для баланса безопасности и удобства
  • Включите login_security для защиты от brute-force атак
  • Регулярно обновляйте RSA ключи (рекомендуется каждые 1-2 года)

  • ["ERROR"] - только критические ошибки
  • [] - отключить файловое логирование
  • true/false - для обратной совместимости
log_file_path string server.log Путь к файлу лога с автоматической ротацией по дате.
  • Абсолютный путь: /var/log/api/server.log
  • Относительный путь: logs/server.log
  • Автоматически создаются файлы с датой: server_2024-01-15.log
  • Автоматически создаются поддиректории
mask_sensitive_data boolean true Маскировать конфиденциальные данные в логах.
  • true - маскировать данные
  • false - не маскировать данные
  • Маскирует: токены, пароли, подписи
  • Заменяет на: t**n, p*****d, s*******e

🎯 Уровни логирования

  • DEBUG - максимальная детализация, включая тела запросов и ответов
  • INFO - основная информация о запросах, подходит для продакшн-среды
  • WARNING - только предупреждения и ошибки
  • ERROR - только критические ошибки

📊 Примеры конфигураций логирования

Минимальная конфигурация (только ошибки):

"logging": { "log_to_db": ["ERROR"], "log_to_file": ["ERROR"], "log_file_path": "server.log", "mask_sensitive_data": true }

Полная конфигурация (все уровни):

"logging": { "log_to_db": ["INFO", "ERROR", "WARNING"], "log_to_file": ["DEBUG", "INFO", "WARNING", "ERROR"], "log_file_path": "/var/log/secure_api/server.log", "mask_sensitive_data": true }

Обратная совместимость (старый формат):

"logging": { "log_to_db": true, // Преобразуется в ["INFO", "ERROR"] "log_to_file": false, // Преобразуется в [] "mask_sensitive_data": true }

Конфигурация CORS

Секция cors определяет политику кросс-доменных запросов (Cross-Origin Resource Sharing), которая контролирует доступ к API с различных доменов.

🌐 Параметры CORS

Параметр Тип По умолчанию Описание Допустимые значения
enabled boolean true Включить обработку CORS заголовков.
  • true - включить CORS
  • false - отключить CORS
  • При false блокируются все кросс-доменные запросы
allowed_origins array ["*"] Список разрешенных доменов.
  • ["*"] - разрешить все домены
  • ["https://example.com"] - конкретный домен
  • ["https://app.com", "https://admin.com"] - несколько доменов
  • Для продакшн указывайте конкретные домены
allowed_methods array ["GET", "POST", "PUT", "DELETE", "OPTIONS"] Разрешенные HTTP методы.
  • GET, POST, PUT, DELETE, OPTIONS
  • PATCH, HEAD - дополнительные методы
  • OPTIONS обязателен для preflight запросов
  • Минимум: ["GET", "POST", "OPTIONS"]
allowed_headers array ["Content-Type", "Token", "Signature"] Заголовки для фактических запросов.
  • Content-Type, Authorization, Token, Signature
  • X-Requested-With, X-CSRF-Token
  • Кастомные заголовки при необходимости
  • Регистронезависимые имена
expose_headers array [] Заголовки, доступные клиенту в JavaScript.
  • Token, Signature - для доступа к заголовкам безопасности
  • Content-Length, X-Request-ID
  • Пустой массив [] - не экспортировать заголовки
allow_credentials boolean false Разрешить отправку cookies и учетных данных.
  • true - разрешить учетные данные
  • false - запретить учетные данные
  • Требует указания конкретных allowed_origins (не "*")
max_age integer 600 Время кэширования preflight запросов.
  • Целое число в секундах
  • Рекомендуемое: 600-3600 секунд
  • Для часто меняющихся настроек: 60-300
  • Максимум: 86400 (24 часа)

🔧 Примеры конфигураций CORS

Разрешить все домены (для разработки):

"cors": { "enabled": true, "allowed_origins": ["*"], "allowed_methods": ["GET", "POST", "PUT", "DELETE", "OPTIONS"], "allowed_headers": ["Content-Type", "Token", "Signature"], "expose_headers": [], "allow_credentials": false, "max_age": 600 }

Строгая политика (для продакшн):

"cors": { "enabled": true, "allowed_origins": ["https://app.company.com", "https://admin.company.com"], "allowed_methods": ["GET", "POST"], "allowed_headers": ["Content-Type", "Token", "Signature"], "expose_headers": ["Token", "Signature"], "allow_credentials": true, "max_age": 3600 }

Аутентификация и авторизация

Система использует двухфакторную аутентификацию, сочетающую статические токены и динамические цифровые подписи с временными метками.

Заголовки безопасности

Каждый защищенный запрос должен содержать два обязательных заголовка:

Token: Bearer your-secret-token
Signature: base64-encoded-rsa-signature

Заголовок Token:

  • Формат: Bearer <token_value>
  • Назначение: Статическая идентификация клиента
  • Требования: Токен должен присутствовать в списке allowed_tokens
  • Пример: Bearer 1234

Заголовок Signature:

  • Формат: Base64-кодированная RSA подпись
  • Назначение: Динамическая верификация запроса и защита от replay-атак
  • Содержимое: Зашифрованная строка формата token.expiry_timestamp
  • Пример: MCwCFAsKB0... (base64 строка)

Ключи безопасности

Для работы системы безопасности необходимы RSA ключи в PEM формате. Сервер использует асимметричное шифрование для верификации запросов и подписи ответов.

Архитектура ключевой пары

🔑 Приватный ключ сервера

  • Назначение: Расшифровка входящих подписей клиентов
  • Формат: PEM, PKCS#1, без пароля
  • Путь: server_private_key_path в конфигурации
  • Безопасность: Должен храниться в защищенном месте на сервере
  • Использование: Для расшифровки подписей клиентов в verify_client_signature()

🔒 Публичный ключ клиента

  • Назначение: Шифрование серверных ответов
  • Формат: PEM, PKCS#1
  • Путь: client_public_key_path в конфигурации
  • Распространение: Предоставляется всем клиентам API
  • Использование: Для шифрования серверных подписей в generate_server_signature()

Процесс работы с ключами

Входящие запросы (клиент → сервер):

  1. Клиент создает подпись: token.expiry_timestamp
  2. Клиент шифрует подпись своим приватным ключом
  3. Сервер расшифровывает подпись приватным ключом сервера
  4. Сервер проверяет валидность токена и времени

Исходящие ответы (сервер → клиент):

  1. Сервер создает подпись: token.expiry_timestamp
  2. Сервер шифрует подпись публичным ключом клиента
  3. Клиент расшифровывает подпись своим приватным ключом
  4. Клиент проверяет валидность ответа

Генерация ключей в Linux

# Создание директории для ключей
mkdir -p keys
cd keys

# Генерация приватного ключа сервера (RSA 2048 бит)
openssl genrsa -out private_server.pem 2048

# Генерация пары ключей для клиента
openssl genrsa -out private_client.pem 2048
openssl rsa -in private_client.pem -pubout -out public_client.pem

# Установка прав доступа
chmod 600 private_server.pem private_client.pem
chmod 644 public_client.pem

Проверка ключей

# Проверка приватного ключа сервера
openssl rsa -in private_server.pem -check -noout

# Проверка приватного ключа клиента
openssl rsa -in private_client.pem -check -noout

# Проверка публичного ключа клиента
openssl rsa -in public_client.pem -pubin -check -noout

# Просмотр содержимого ключей
openssl rsa -in private_server.pem -text -noout
openssl rsa -in public_client.pem -pubin -text -noout

Распределение ключей

🔐 Приватный ключ сервера

  • Файл: private_server.pem
  • Хранится у: Только на сервере
  • Передается: Никогда
  • Назначение: Расшифровка входящих подписей клиентов

🔑 Публичный ключ клиента

  • Файл: public_client.pem
  • Хранится у: Сервер + Клиенты
  • Передается: Распространяется всем клиентам API
  • Назначение: Шифрование исходящих ответов сервера

🔒 Приватный ключ клиента

  • Файл: private_client.pem
  • Хранится у: Только у клиента
  • Передается: Никогда
  • Назначение: Создание подписей для исходящих запросов

🔄 Процесс работы с ключами

Входящие запросы (проверка подписи):

  1. Клиент создает подпись:
    token.expiry_timestamp → Шифрование приватным ключом клиента → Base64
  2. Сервер проверяет (verify_client_signature):
    Base64 → Расшифровка приватным ключом сервера → Проверка token и expiry_timestamp

Исходящие ответы (создание подписи):

  1. Сервер создает подпись (generate_server_signature):
    token.expiry_timestamp → Шифрование публичным ключом клиента → Base64
  2. Клиент проверяет ответ:
    Base64 → Расшифровка приватным ключом клиента → Проверка данных

📁 Конфигурационные пути

  • server_private_key_path: keys/private_server.pem - Приватный ключ сервера для расшифровки
  • client_public_key_path: keys/public_client.pem - Публичный ключ клиента для шифрования ответов

🛡️ Важные особенности реализации

  • Проверка подписи: Используется padding.PKCS1v15() для совместимости с PHP openssl_public_encrypt
  • Формат данных: token.expiry_timestamp (пример: "abc123.1690000000")
  • Время жизни: Контролируется signature_ttl (по умолчанию 300 секунд)
  • Отключение безопасности: Через параметры disable_certificates, disable_token_auth, disable_signature
  • Асинхронная загрузка: Ключи загружаются через run_in_executor для избежания блокировки

⚠️ Важные замечания по ключам

  • Приватные ключи никогда не передаются между сервером и клиентами
  • Сервер использует свой приватный ключ только для расшифровки входящих запросов
  • Сервер использует публичный ключ клиента для шифрования исходящих ответов
  • Регулярно обновляйте ключи (рекомендуется каждые 6-12 месяцев)
  • Используйте разные пары ключей для разных сред (dev/staging/prod)
  • Храните бэкапы ключей в защищенном месте
  • Для продакшн-среды используйте ключи длиной 2048 бит или больше
